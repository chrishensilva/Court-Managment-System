Executive Summary

The Court Case Management System (Court CMS) is a cloud-based, multi-tenant SaaS application designed to help law firms and legal practitioners manage their case portfolio, lawyers, client records, and court schedules efficiently. The platform operates as a fully hosted solution — the frontend is deployed on Vercel and the backend on Render, backed by an Aiven-hosted MySQL database.

The system was built to solve a critical operational problem for small to mid-sized law firms: the management of multiple active cases, assignment of lawyers, tracking of court dates, and document management — all from a single, secure, multi-user interface accessible from any device.

2.4 Multi-Tenancy Model

The platform uses a shared-table, discriminator-key multi-tenancy model. Every database record carries an owner_id foreign key tied to the authenticated admin user's ID. This ensures strict data isolation at the application layer — each tenant (law firm / admin user) sees only their own cases, lawyers, and settings. All SQL queries are filtered by owner_id at the backend before any data is returned to the client.

ℹ️
The multi-tenancy model allows the platform to scale horizontally (add more admin accounts) without any structural database changes. Each admin user acts as an independent organisational root owning all resources created under them.

3.5 Authentication Flow

The system uses a dual-mechanism JWT strategy designed for cross-domain deployments:

  1. User submits credentials → server validates with bcrypt → signs a JWT (RS/HS256, configurable expiry).
  2. Token is set as httpOnly, Secure, SameSite=None cookie (cross-domain) AND returned in the response body.
  3. Frontend stores the token in localStorage and attaches it as Authorization: Bearer <token> on every API request.
  4. Backend middleware checks the cookie first; if absent, validates the Authorization header — ensuring compatibility across all browser environments.
  5. On logout: cookie is cleared server-side + localStorage is cleared client-side.

3.6 Email Architecture

Each admin tenant configures their own Gmail SMTP credentials (stored AES-256-CBC encrypted in the database). When a lawyer is assigned to a case, the system dynamically creates a Nodemailer transporter using the tenant's credentials and dispatches a styled HTML email. The system is configured with family: 4 to force IPv4 (required for Render hosting), with a 15-second connection timeout.
